## 1. 正常取消场景

### TC_CANCEL_001: AI 响应前的正常取消

| 属性 | 值 |
|------|-----|
| 用例编号 | TC_CANCEL_001 |
| 用例标题 | AI 响应前的正常取消 |
| 测试类型 | 功能测试 |
| 优先级 | **P0** (阻塞) |
| 关联需求 | REQ_AI_CHAT_003 |
| 发现日期 | 2026-04-06 |

**测试前提**
- Tauri 应用运行中
- AI 引擎已构建
- 网络连接正常

**测试步骤**
1. 在对话框输入 "你好"
2. 点击发送按钮
3. 在 AI 响应前（看到 "思考中" 或 "加载中" 状态时）立即点击取消按钮

**预期结果**
- 对话应该立即停止
- 不应该显示 "思考中" 或 "加载中" 状态超过 2 秒
- 输入框应该立即恢复可用状态
- 不应该发送任何请求到 AI 后端（可通过日志验证）

**测试数据**
- 输入消息: "你好"
- 预期取消响应时间: < 500ms

**测试方式**: 手动测试

## 2. 响应中取消场景

### TC_CANCEL_003: AI 开始响应后的取消

| 属性 | 值 |
|------|-----|
| 用例编号 | TC_CANCEL_003 |
| 用例标题 | AI 开始响应后的取消 |
| 测试类型 | 功能测试 |
| 优先级 | **P0** (阻塞) |
| 关联需求 | REQ_AI_CHAT_003 |

**测试前提**
- Tauri 应用运行中
- 网络连接正常
- 使用长响应请求（如 "讲一个很长的故事，至少1000字"）

**测试步骤**
1. 在对话框输入 "讲一个很长的故事，至少1000字"
2. 点击发送按钮
3. 等待 AI 开始响应（看到第一个字符或句子出现）
4. 立即点击取消按钮

**预期结果**
- 当前响应应该立即停止
- 已显示的部分内容应该保留在对话中
- 不应该继续显示新的内容
- 输入框应该恢复可用状态
- 后续请求应该不受影响

**测试数据**
- 输入消息: "讲一个很长的故事，至少1000字"
- 预期已显示内容保留

**测试方式**: 手动测试

- 可以继续正常对话

**测试数据**
- 输入: 长篇内容请求

**测试前提**
- Tauri 应用运行中
- 配置一个不存在的 API 地址或断网环境

**测试步骤**
1. 在对话框输入 "测试"

## 4. 并发请求取消

### TC_CANCEL_008: 快速连续发送和取消

| 属性 | 值 |
|------|-----|
| 用例编号 | TC_CANCEL_008 |
| 用例标题 | 快速连续发送和取消 |
| 测试类型 | 压力测试 |
| 优先级 | **P0** (阻塞) |

**测试前提**
- Tauri 应用运行中
- 网络连接正常

**测试步骤**
1. 在对话框输入 "消息1"
2. 点击发送按钮
3. 立即再次输入 "消息2" 并点击发送
4. 立即再次输入 "消息3" 并点击发送
5. 快速点击取消按钮 3 次

**预期结果**
- 不应该出现多个并发请求
- 只有一个请求被处理
- 已发送的请求应该被正确取消
- 不应该出现请求队列混乱
- 输入框应该保持可用

**测试数据**
- 快速发送消息: "消息1", "消息2", "消息3"
- 预期: 最多 1 个请求被处理

**测试方式**: 手动测试

## 5. UI 状态正确性

### TC_CANCEL_010: 取消按钮状态验证

| 属性 | 值 |
|------|-----|
| 用例编号 | TC_CANCEL_010 |
| 用例标题 | 取消按钮状态验证 |
| 测试类型 | UI 测试 |
| 优先级 | **P0** (阻塞) |

**测试前提**
- Tauri 应用运行中

**测试步骤**
1. 打开对话框界面
2. 观察初始状态下取消按钮是否可见/可用
3. 发送一条消息
4. 观察发送按钮和取消按钮的状态变化

**预期结果**
- 初始状态: 取消按钮应该隐藏或禁用
- 发送后: 取消按钮应该显示/可用
- 取消后: 取消按钮应该隐藏或恢复禁用状态
- 按钮状态切换应该流畅，无闪烁

**测试方式**: 手动测试
