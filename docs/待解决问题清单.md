## 问题 7：编辑器空格宽度问题

### 问题描述

用户反馈按空格键时，空格宽度不等于一个汉字位置。用户需要的是**按一次空格键，插入的空格字符总宽度 = 一个汉字位置**。

### 影响范围

- 所有使用空格键进行缩进的用户
- 中文写作场景

### 根因分析

之前的解决方案使用了 CSS `letterSpacing` 属性，但这会影响**所有字符**的间距，而不是只控制空格宽度。

### 解决方案

✅ 已完成修复：

**新的实现方案**：拦截空格键，动态插入多个空格字符

1. 修改 `firstLineIndent.ts`：
   - 移除 CSS `letterSpacing` 的使用
   - 添加空格键拦截逻辑
   - 按空格键时计算并插入正确数量的空格字符
   - 计算公式：`spaceCount = 1 / spaceWidthRatio`

2. 计算逻辑：
   - `spaceWidthRatio = 1.0`（100%）→ 按一次空格 = 插入 1 个空格
   - `spaceWidthRatio = 0.5`（50%）→ 按一次空格 = 插入 2 个空格
   - 首行缩进：`indentChars / spaceWidthRatio` 个空格

3. 修改 `editorTheme.ts`：
   - 移除 `.cm-content` 的 `letterSpacing` 设置
   - 避免影响其他字符的间距

### 验收标准

- [x] 按一次空格键，插入的空格总宽度 = 一个汉字位置（当 spaceWidthRatio=1.0）
- [x] 首行缩进正确计算空格数
- [ ] 字符间距不受影响

### 优先级

- P0（紧急）- 核心功能缺陷

## 问题 8：保存状态指示不准确

### 问题描述

用户反馈右下角保存指示灯一直显示"未保存"状态，即使实际上章节已经自动保存成功。这种状态不一致会导致用户困惑和焦虑。

### 影响范围

- 所有使用 Creator Studio 的用户
- 特别影响关心数据安全的用户
- 可能导致用户频繁手动保存，影响写作体验

### 根因分析

1. **状态转换逻辑问题**：`useChapterManager` 中的 `saveStatus` 可能在某些场景下没有正确更新
2. **状态同步问题**：`StatusBar` 组件接收的 `saveStatus` 与 `useChapterManager` 中的状态可能不同步
3. **初始化问题**：章节切换时状态可能未正确重置

### 期望体验

1. 保存状态准确反映实际保存状态
2. 状态切换流畅，无闪烁或跳跃
3. 保存完成后立即更新为"已保存"状态
4. 保存失败时明确显示错误信息

### 建议解决方案

1. **状态机规范化**：
   - 定义明确的状态转换规则
   - 确保所有状态变更都经过同一入口

2. **添加状态转换日志**（调试用）：
   ```typescript
   // 临时添加，用于调试
   console.debug(`[SaveStatus] ${prevStatus} -> ${newStatus}`, context);
   ```

3. **确保事件正确广播**：
   - 检查 `creatorai:saveStatus` 事件是否正确触发
   - 验证 `StatusBar` 是否正确监听事件

4. **添加状态同步检查**：
   - 在 `MainLayout` 中添加调试信息
   - 确保 `chapter.saveStatus` 正确传递给 `StatusBar`

### 验收标准

- [ ] 右下角保存指示灯初始状态为"已保存"
- [ ] 编辑内容后状态正确切换为"未保存"
- [ ] 自动保存触发时状态显示"保存中..."
- [ ] 保存完成后状态切换回"已保存"
- [ ] 状态切换响应时间 < 100ms

### 优先级

- P0（紧急）- 核心功能缺陷，用户体验严重受影响
